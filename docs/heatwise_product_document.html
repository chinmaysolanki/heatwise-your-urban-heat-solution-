What is HeatWise?

HeatWise is an AI-powered mobile application that helps urban residents and property owners combat the heat island effect by designing, installing, and monitoring green cover on rooftops, terraces, and balconies. Users scan their space with a phone camera, receive ML-generated plant species recommendations, connect with verified local installers, and track the resulting carbon offset — all within a single seamless journey.

Core Value Proposition

Every 10 m² of green cover installed drops surface temperature by ~0.67°C, saves 5–15% in air conditioning energy, and sequesters ~0.21 kg CO₂ per year. HeatWise makes this measurable, actionable, and verifiable at scale.

05 Recommendation Engine — Deep Dive

Engine Architecture (3-Tier Hybrid)

Design Principle

The engine always returns a result. If the ML model is unavailable, the catalog hybrid fallback fires. If the catalog is incomplete, the rules-only fallback guarantees at least 3 ranked candidates. Every tier produces the same output shape.

// Full recommendation pipeline — lib/recommendation/orchestrateLayoutRecommendations.ts POST /api/recommendations/generate │ ├─ buildRecommendationGenerateRequestFromPhotoSession() │ photoSession → ProjectInput → RecommendationGenerateRequest │ ├─ enrichRecommendationsWithPricing() // cost per candidate ├─ buildGeoEnrichmentBundle() // lat/lon → microclimate │ ├─ // TIER 1: ML (Python sidecar via HTTP) │ mlRecommendationService.predict() │ → species scores (multi-label classifier) │ → ranking model scores │ ├─ // TIER 2: Catalog Hybrid Fallback (if ML unavailable) │ catalogHybridFallback() │ → HARD_PET_UNSAFE_SPECIES exclusion │ → waterScarcityHardExclude (drought-tolerant check) │ → petSafeRankingNudge / reconcileChildPetSafety │ → score candidates by environment match │ ├─ // TIER 3: Rules-Only Fallback │ rulesOnlyFallback() // guaranteed 3+ results │ ├─ layoutEligibility() // north wall / perimeter / center ├─ mergeMlIntoLayoutRecommendations() // combine scores + layout ├─ recommendationRolloutGuardrails() // A/B experiment safety ├─ attachGeoExplanationToCandidates() // "why this species" └─ scheduleRecommendationRuntimeObservation() // async telemetry

Species Scoring Logic

Each species is scored on a weighted combination of environment match signals. The catalog hybrid fallback uses these rules:

Sun Exposure Match
Compares species sunRequirement (shade / partial / full) against the derived sunExposure from UV index or heat exposure. Full sun species are penalised in shaded environments and vice versa.
Wind Tolerance
windExposure (sheltered / moderate / windy / severe) mapped to windLevel (low / medium / high). Species with low wind tolerance are excluded when windExposure ≥ windy.
Water Scarcity Hard Exclusion
If scenario has scarce water: species with droughtTolerant=false AND unknown effective water demand are hard-excluded. Prevents recommending water-hungry plants in arid contexts.
Pet Safety Hard Exclusion
If child_pet_safe_required=1: species with dbPetSafe=false are hard-excluded before ranking. ASPCA toxicity data embedded in seed. Lemongrass and mint are correctly flagged as pet-unsafe (ASPCA-confirmed).
Space Type Filtering
spaceType (rooftop / terrace / balcony) filters out species with incompatible height, root depth, or weight requirements. Balcony scenarios exclude large-root species automatically.
Layout Zone Eligibility
layoutEligibility.ts assigns each species to valid placement zones: north wall (tall climbers), perimeter (medium shrubs), center fill (ground cover). Returns eligibility codes per zone with failure reasons.

A/B Experiment & Rollout System

AssignmentPOST /api/recommendations/assignment returns experiment variant (control / challenger) per user session.
GuardrailsrecommendationRolloutGuardrails.ts checks: minimum species count, maximum exclusion rate, diversity score floor. Rolls back to control if challenger fails guardrails.
Shadow EvalEvery recommendation run optionally runs appendRuntimeEvaluation() which scores the challenger variant without serving it to the user. Results stored in RecommendationRuntimeObservation.
TelemetryRecommendationTelemetrySession (immutable per request) + RecommendationCandidateSnapshot (exact card shown) + RecommendationTelemetryEvent (user signals). All append-only — never updated.

06 ML Pipeline & AR System
8 / 12

ML Model Architecture

ClassifierSpecies multi-label classifier (scikit-learn). Input features: spaceType, sunExposure, windLevel, floorLevel, budgetRange, maintenanceLevel, lat/lon. Output: probability scores for each species code in the catalog.
RankerRanking model trained on telemetry: which species the user selected given the ranked list. Optimised for NDCG. Outputs a re-ranking score combined with classifier probability.
ServingPython Flask/FastAPI sidecar in /ml/serving/. TypeScript calls it via HTTP using lib/services/mlRecommendationService.ts. Config via env vars: HEATWISE_ML_CWD, HEATWISE_REGISTRY_DIR, HEATWISE_SPECIES_CSV.
Registryml/registry/ stores versioned model artefacts. Each version includes: model weights, feature schema, evaluation metrics, training dataset hash. Enables safe rollback.

Training Data Pipeline

1. Telemetry Collection
Every recommendation run creates a RecommendationTelemetrySession. User selections, impressions, and feedback logged as RecommendationTelemetryEvent rows. All immutable and append-only.
2. Outcome Verification
Installed projects produce VerifiedInstallRecord after installer submits photos. Follow-up check-ins at 30/90/180 days provide survival and cooling delta data as ground truth labels.
3. Export Pipeline
npm run ml:export-telemetry-pipeline exports labelled training data from Prisma to CSV files in ml/data/. Feature engineering applied: derive heat level from temp, encode categoricals.
4. Model Training
ml-models/scripts/train_species_model.py runs scikit-learn training. Validated against held-out test set. Metrics (precision@k, NDCG, recall) logged to registry.
5. Deploy & Shadow Eval
New model deployed to registry. Rollout guardrails activate shadow evaluation: new model runs alongside control without affecting user output. If guardrails pass over N requests, traffic shifted to new model.

AR Measurement System

Filesrc/live-ar/screens/LiveARMeasurementScreen.tsx — primary AR screen.
Perspective MathscreenToWorld() uses full ray-floor intersection. Given a tap (sx, sy) in screen space, device pitch β and roll γ, image width W and height H, derives a unit direction vector and intersects with y=0 floor plane to get real-world (x, z) coordinates.
Live ModeDeviceOrientationEvent provides beta (pitch) and gamma (roll) in real time. Tapping places corner markers. After 4 corners, area = |cross product| / 2 × 2 for rectangle approximation.
Gallery ModeStatic photo has no device orientation. getContainedImageBounds() computes objectFit:contain letterbox offset from naturalWidth/Height vs clientWidth/Height. galleryBeta state (default 60°) substitutes for live pitch. 4 preset buttons: Overhead (β=20), High (β=42), Mid (β=60), Low (β=76).
Canvas CompatrRect() helper: uses native ctx.roundRect() if available (Chrome 99+, Safari 15.4+, Firefox 112+), falls back to quadraticCurveTo() for older Android WebViews. All 8 call sites use this helper.
OverlayCanvas drawn on top of camera/image. Corner markers, edge lines, area label, tip banner, mini-map all drawn with requestAnimationFrame loop. DM Sans font loaded for labels.

Production Deployment

DatabasePostgreSQL in production. Migrations via npx prisma migrate deploy in CI/CD pipeline before new version deploy.
ML ServingPython sidecar deployed as a separate service (Cloud Run or Railway). HEATWISE_ML_CWD and HEATWISE_REGISTRY_DIR point to cloud storage paths.
Android Buildnpm run build → cap sync android → Android Studio release build → signed APK/AAB → Google Play Store. CAP_SERVER_URL points to production Vercel URL.
CORSNext.js allows 10.0.2.2 (emulator loopback) and localhost by default in dev. Production origins locked to Vercel URL. Additional origins via HEATWISE_DEV_EXTRA_ORIGINS.
Key Architectural Decisions

Monorepo: All code in one repo reduces integration friction between frontend, API, and ML.
Pages Router: Chosen over App Router for stability with NextAuth and the large existing component tree.
Inline styles: Eliminates CSS class name collisions in the single-file component architecture of HeatWiseApp.jsx.
Append-only telemetry: Prevents data loss and enables reliable ML retraining on immutable historical data.
3-tier recommendation fallback: Guarantees a result is always returned, even when ML serving is unavailable.
